# JJC 任务生命周期规则

本文档定义 edict PoC 当前使用的 `JJC` 任务编号、创建条件、续办规则、流转节点与群回报约定。

## 1. 任务号格式

当前任务号采用：

```text
JJC-YYYYMMDD-XXX
```

例如：
- `JJC-20260308-001`

### 字段含义
- `JJC`：任务前缀
- `YYYYMMDD`：任务首次立项日期
- `XXX`：当天顺序号，从 `001` 起递增

## 2. 什么情况下新建任务号

只有当一条消息被判定为 **正式事项型**，且不属于任何已有事项的续办时，才新建任务号。

### 正式事项型常见特征
- 明确说“正式事项 / 正式任务 / 写入看板 / 转中书省 / 纳入流程 / 立项 / 推进”
- 或语义上明显需要多步骤推进、后续跟踪、正式产出

### 新建时应做的动作
1. 生成新的 `JJC-...`
2. 写入 weiUI
3. 初始化流转
4. 在群里回报任务号、当前阶段、下一步

## 3. 什么情况下必须复用任务号

当一条消息被判定为 **续办型** 时，必须优先复用既有任务号，不允许重复立项。

### 续办型常见特征
- 显式包含已有任务号，如 `JJC-...`
- 出现“继续 / 接着 / 更新进度 / 汇报状态 / 基于上一条继续 / 拆成阶段”
- 回复/引用的是已存在任务消息

### 续办时应做的动作
1. 绑定原任务号
2. 更新原 weiUI 条目
3. 追加原流转
4. 在群里回报该任务的新进展

## 4. 流转节点建议

当前建议的最小流转节点包括：

1. 宰相收件
2. 转中书省
3. 中书省起草 / 处理中
4. 待补充信息（如有）
5. 已完成 / 已回报

项目可根据需要继续细化，但应保持：
- 节点语义清楚
- 群里与 weiUI 的状态一致
- 同一任务不要因为不同表述而产生多个并列状态

## 5. 群内回报约定

### 新建任务时
建议回报：
- 任务号
- 当前阶段
- 下一步

示例：

```text
已按正式事项处理。
任务号：JJC-20260308-002
当前进度：已登记看板，已转中书省。
下一步：等待中书省起草第一版方案。
```

### 续办任务时
建议回报：
- 原任务号
- 本轮新增进展
- 下一步

示例：

```text
JJC-20260308-001 已继续推进。
当前新增进展：已拆成三个执行阶段。
下一步：进入第一阶段的只读审计准备。
```

## 6. weiUI 对应关系

### 允许进入 weiUI 的任务
- 正式事项型
- 续办型

### 不应进入 weiUI 的消息
- 问答型
- 自我介绍
- 架构说明
- 普通解释/讨论
- 闲聊

## 7. 设计原则

1. 宁可少建任务，也不要乱建任务
2. 续办优先复用任务号，不重新立项
3. 群里回复与 weiUI 状态尽量保持一致
4. 任务流转要可追踪、可回读、可继续推进

## 8. 当前阶段说明

本规则是基于元老院群 + 柏拉图 + weiUI 联动的实际使用经验整理出的落地规则，属于对原有 edict 大架构的运行层补充。
