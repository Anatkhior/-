# 群消息分类与看板写入策略

本文档定义元老院群入口的消息分类规则，以及哪些消息应写入 weiUI，哪些不应写入。

## 1. 适用范围

当前规则适用于：
- channel = feishu
- chat_type = group
- 群：元老院
- 消息明确 @ 了 bot

## 2. 分类顺序

所有命中上述范围的消息，按以下顺序判断：

1. 续办型
2. 正式事项型
3. 问答型

这个顺序不可颠倒。

## 3. 续办型

### 定义
不是新开事项，而是在继续推进已有任务。

### 触发特征
- 明确包含 `JJC-...`
- 出现：继续 / 接着 / 更新进度 / 汇报状态 / 基于上一条继续 / 拆成阶段
- 回复或引用了已存在任务消息

### 执行动作
- 不新建任务号
- 复用既有任务号
- 更新原 weiUI 条目
- 追加原流转
- 视复杂度决定是否转柏拉图 / 中书省继续处理

## 4. 正式事项型

### 定义
需要多步骤推进、后续跟踪、正式产出或系统内持续处理的事项。

### 触发特征
- 明确说：正式事项 / 正式任务 / 写入看板 / 转中书省 / 纳入流程 / 立项 / 推进
- 或语义上明显要求：
  - 分阶段执行
  - 形成方案
  - 持续跟进
  - 输出正式结果

### 执行动作
- 新建任务号（JJC-...）
- 写入 weiUI
- 初始化流转
- 按需要转柏拉图 / 中书省处理

## 5. 问答型

### 定义
既不是续办型，也不是正式事项型的消息。

### 常见场景
- 自我介绍
- 职责说明
- 架构解释
- 普通问答
- 普通判断与讨论
- 闲聊

### 执行动作
- 直接回答
- 不新建任务号
- 不写 weiUI
- 不开启正式流转

## 6. weiUI 写入门槛

### 必须允许写入的情况
- 正式事项型
- 续办型

### 必须禁止写入的情况
- 问答型
- 自我介绍
- 架构说明
- 普通解释/讨论
- 闲聊

## 7. 不确定时的策略

如果系统不能清晰判断一条消息是问答型还是正式事项型，应优先：

1. 先澄清
2. 不要立刻写 weiUI
3. 不要乱建任务号

推荐澄清方式示例：

```text
这条我可以按普通问答处理，也可以按正式事项写入看板推进。你要我按哪种方式处理？
```

## 8. 目标

本策略的目标是：
- 防止普通聊天污染看板
- 防止同一事项重复立项
- 保证群聊与 weiUI 的状态尽量一致
- 让柏拉图和元老院入口更像一个制度化中枢，而不是随意聊天助手

## 9. 当前阶段说明

这份策略文档用于补齐原有 edict 架构文档中未细化的运行层规则，属于元老院群入口与 weiUI 联动的实际落地补充。
