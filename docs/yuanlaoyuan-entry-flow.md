# 元老院群入口工作流说明

本文档说明当前 edict PoC 中，飞书群「元老院」作为外部入口时的工作流设计。

## 1. 入口背景

当前系统已接入一个独立的 Feishu bot，服务于飞书群：

- 群名：元老院
- chat_id：`oc_26487f6c792f445d4450570ddcef97a2`

该群当前作为 edict PoC 的对外入口之一，用于：
- 问答
- 正式事项发起
- 已有事项续办
- 结果回报

## 2. 核心角色

### 柏拉图（taizi）
- 身份：元老院宰相
- 对外职责：
  - 日常问答直接处理
  - 正式事项整理后转中书省推进
  - 将结果向奥古斯都回报

### 元老院前台代理（main）
- 身份：入口代理层
- 对外职责：
  - 接住群里的消息
  - 在必要时进行分类、转办、兜底
  - 尽量不抢柏拉图的人设

## 3. 消息分类规则

元老院群消息分为三类：

### A. 续办型
#### 触发条件
- 明确包含 `JJC-...`
- 出现“继续 / 接着 / 更新进度 / 汇报状态 / 基于上一条继续 / 拆成阶段”
- 或回复/引用的是已存在任务消息

#### 动作
- 不新建任务号
- 复用既有任务号
- 更新原 weiUI 条目
- 追加原流转

### B. 正式事项型
#### 触发条件
- 明确说“正式事项 / 正式任务 / 写入看板 / 转中书省 / 纳入流程 / 立项 / 推进”
- 或语义上明显需要多步骤推进、后续跟踪、正式产出

#### 动作
- 新建任务号（JJC-...）
- 写入 weiUI
- 初始化流转
- 转柏拉图 / 中书省

### C. 问答型
#### 定义
- 既不是续办型，也不是正式事项型

#### 动作
- 直接回答
- 不新建任务号
- 不写 weiUI
- 不启动正式流转

## 4. weiUI 写入门槛

### 允许写入
- 正式事项型
- 续办型

### 禁止写入
- 问答型
- 自我介绍
- 架构说明
- 普通解释/讨论
- 闲聊

## 5. 对外回复原则

### 柏拉图
- 优先保持“元老院宰相”身份
- 使用现代中文
- 对用户称呼“奥古斯都”
- 对外尽量保持单一窗口感，不暴露内部复杂流转

### 前台代理
- 低存在感
- 不抢人格
- 仅在超时、失败、空结果时兜底

## 6. 当前工作目标

当前这套设计的目标不是一次性完成最终形态，而是逐步实现：

1. 元老院群稳定可用
2. 柏拉图人格稳定输出
3. weiUI 与群聊任务流一致
4. 正式事项与续办不乱串号
5. 后续再加强失败路径与多人群鲁棒性
